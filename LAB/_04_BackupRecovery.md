# 1. Цель лабораторной работы

Получить практические навыки организации резервного копирования данных серверной инфраструктуры, автоматизации процесса бэкапа и восстановления данных после сбоя.

---

# 2. Задачи лабораторной работы

Студент должен:

- развернуть отдельный Backup Server;
- настроить SSH-доступ между серверами;
- организовать резервное копирование файлов с использованием rsync;
- реализовать резервное копирование базы данных MySQL;
- автоматизировать процесс через cron;
- смоделировать потерю данных;
- выполнить восстановление из резервной копии.

---

# 3. Теоретические сведения

## 3.1 Виды резервного копирования

**Full backup** — полная копия данных.  
**Incremental backup** — копируются только изменения с момента последнего бэкапа.  
**Differential backup** — копируются изменения с момента последнего полного бэкапа.

---

## 3.2 Используемые инструменты

- rsync  
- mysqldump  
- cron  
- SSH  
- sudo

# 6. Ход работы

---

## Шаг 1. Подготовка Backup Server

```bash
hostnamectl set-hostname backup.studentXX.local
apt update && apt upgrade -y
apt install rsync openssh-client -y
````

---

## Шаг 2. Настройка SSH-доступа

```bash
ssh-keygen
ssh-copy-id user@192.168.XX.10
ssh-copy-id user@192.168.XX.20
ssh-copy-id user@192.168.XX.30
```

Проверка подключения:

```bash
ssh user@192.168.XX.10
```

---

## Шаг 3. Создание каталога хранения

```bash
mkdir -p /backup/dns
mkdir -p /backup/mail
mkdir -p /backup/zabbix
```

---

## Шаг 4. Резервное копирование файлов

Пример копирования:

```bash
rsync -avz user@192.168.XX.10:/etc /backup/dns/
rsync -avz user@192.168.XX.20:/var/mail /backup/mail/
```

---

## ВАЖНО: Права доступа при резервном копировании

При копировании системных каталогов (`/etc`, `/var/spool`, `/var/lib/mysql`)
обычный пользователь может не иметь достаточных прав.

Возможные решения:

### 1. Использование sudo локально

```bash
sudo rsync -avz user@192.168.XX.10:/etc /backup/dns/
```

### 2. Использование sudo на удалённой стороне

```bash
rsync -a --rsync-path="sudo rsync" user@192.168.XX.10:/etc /backup/dns/
```

### 3. Запуск скрипта от root

```bash
sudo /usr/local/bin/backup.sh
```

Без прав администратора:

* возможны ошибки `Permission denied`;
* часть файлов не будет скопирована;
* резервная копия окажется неполной.

---

## Шаг 5. Резервное копирование базы данных

На Mail сервере:

```bash
mysqldump -u root -p maildb > /tmp/maildb.sql
```

На Backup Server:

```bash
rsync -avz user@192.168.XX.20:/tmp/maildb.sql /backup/mail/
```

---

## Шаг 6. Создание автоматического скрипта

```bash
nano /usr/local/bin/backup.sh
```

Содержимое:

```bash
#!/bin/bash

DATE=$(date +%F)
mkdir -p /backup/$DATE

rsync -a --rsync-path="sudo rsync" user@192.168.XX.10:/etc /backup/$DATE/dns
rsync -a --rsync-path="sudo rsync" user@192.168.XX.20:/var/mail /backup/$DATE/mail
rsync -a --rsync-path="sudo rsync" user@192.168.XX.30:/etc/zabbix /backup/$DATE/zabbix
```

Сделать исполняемым:

```bash
chmod +x /usr/local/bin/backup.sh
```

---

## Шаг 7. Автоматизация через cron

```bash
crontab -e
```

Добавить:

```
0 2 * * * sudo /usr/local/bin/backup.sh
```

---

## Шаг 8. Имитация аварии

На Mail сервере:

```bash
sudo rm -rf /var/mail/*
```

---

## Шаг 9. Восстановление

```bash
rsync -av /backup/DATE/mail/ user@192.168.XX.20:/var/mail/
```

Проверить целостность данных.
